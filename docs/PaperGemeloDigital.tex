    {\fontsize{12}{14}\selectfont\bfseries Resumen}
\end{center}

Este artículo presenta la arquitectura computacional de un Gemelo Digital Fenomenológico desarrollado en Python, cuyo diseño prioriza la modularidad, la validación en múltiples capas y la replicabilidad en diferentes industrias de proceso. El sistema se organiza en tres módulos independientes y acoplados: un núcleo de balances fenomenológicos (\texttt{stage\_equations.py}), un motor de restricciones de capacidad (\texttt{equipment\_constraints.py}) y una interfaz de sala de control en Streamlit (\texttt{app.py}). Las constantes de equipo, los límites de control y los factores de seguridad están externalizados en archivos de configuración con validación automática de rangos físicos. Un conjunto de pruebas unitarias verifica la integridad del sistema ante modificaciones. Como caso de validación, se instanció la arquitectura para una planta de Proteína Aislada de Soya (ISP) de $1000\,\text{kg/h}$. Una campaña de Monte Carlo sobre el diseño base inicial arrojó solo $4.70\%$ de éxito operativo; tras redimensionar los equipos críticos mediante la arquitectura propuesta, una segunda campaña de 300,000 corridas alcanzó $99.88\%$ de éxito ($4.53\sigma$), validando la utilidad del enfoque para diagnóstico y optimización de procesos.

\section{Introducción}

El Gemelo Digital (DT) se ha consolidado como una herramienta clave en la transición hacia la Industria 4.0 y 5.0 [1]. Sin embargo, gran parte de la literatura se enfoca en casos de estudio específicos sin presentar una arquitectura de código replicable. Este trabajo aborda esa brecha describiendo la estructura computacional de un DT fenomenológico diseñado para ser adaptado a diferentes procesos con mínimo esfuerzo.

En la literatura técnica [2] se diferencian tres niveles de integración virtual: \textbf{Modelo Digital} (representación estática), \textbf{Sombra Digital} (flujo unidireccional de datos) y \textbf{Gemelo Digital} (flujo bidireccional con control predictivo). El sistema aquí presentado corresponde a un Gemelo Digital fenomenológico: su núcleo de cálculo se basa en ecuaciones de conservación de masa y energía, no en correlaciones empíricas o modelos de caja negra. Esta elección permite que el modelo sea interpretable, depurable y transferible entre procesos.

La contribución principal de este artículo es la arquitectura de software del DT, compuesta por tres módulos independientes que pueden ser reemplazados o extendidos individualmente. La sección~2 detalla la estructura de cada módulo, los patrones de validación y el sistema de pruebas. La sección~3 describe los requisitos de replicabilidad. La sección~4 presenta los resultados de instanciar la arquitectura para una planta de Proteína Aislada de Soya (ISP).

\section{Arquitectura del Gemelo Digital}

El sistema se organiza en tres módulos principales y dos archivos de configuración, todos desarrollados en Python 3.10+. La comunicación entre módulos se realiza exclusivamente mediante diccionarios con estructura definida, lo que permite reemplazar cualquier módulo sin afectar los demás.

\subsection{Vista General de Módulos}

\begin{itemize}
    \item \texttt{core/equipment\_specs.py} -- Configuración de parámetros de equipo y límites de capacidad. Define los valores nominales (\texttt{EQUIPMENT\_SPEC\_DEFAULTS}), los rangos físicos permisibles (\texttt{EQUIPMENT\_SPEC\_LIMITS}) y las funciones de validación. Cualquier adaptación a un nuevo proceso comienza modificando este archivo.
    \item \texttt{core/stage\_equations.py} -- Núcleo fenomenológico. Contiene las ecuaciones de balance de materia y energía para cada etapa del proceso, organizadas como funciones independientes. Define \texttt{CONTROL\_LIMITS} (rangos válidos de las variables de control) y \texttt{KPI\_SPECS} (rangos objetivo de los indicadores de calidad). La función \texttt{run\_process\_model} orquesta la ejecución secuencial de todas las etapas.
    \item \texttt{core/equipment\_constraints.py} -- Motor de restricciones. Evalúa las variables de proceso contra las capacidades físicas del equipamiento y genera alarmas cuando se superan los umbrales de seguridad.
    \item \texttt{app.py} -- Interfaz de sala de control en Streamlit. Integra el bucle de inercia dinámica y la visualización en tiempo real de KPIs, alarmas y diagramas de flujo.
    \item \texttt{tests/} -- Pruebas unitarias que verifican la consistencia del balance de materia, los KPIs del caso base y la detección correcta de cada restricción de capacidad.
\end{itemize}

\subsection{Configuración Externalizada y Validación}

Todos los parámetros del proceso están externalizados en constantes de módulo con validación automática. El archivo \texttt{equipment\_specs.py} define tres capas de configuración:

\begin{enumerate}
    \item \textbf{Valores nominales} (\texttt{EQUIPMENT\_SPEC\_DEFAULTS}): Capacidades de diseño recomendadas para operación normal.
    \item \textbf{Límites físicos} (\texttt{EQUIPMENT\_SPEC\_LIMITS}): Rangos [mín, máx] que garantizan que los parámetros sean físicamente realizables. La función \texttt{validate\_equipment\_specs} verifica que cada valor esté dentro de su rango.
    \item \textbf{Límites operativos} (\texttt{CAPACITY\_LIMIT\_DEFAULTS}): Factores de seguridad (fracciones máximas de llenado, carga térmica, etc.) que definen el margen operativo antes de disparar una alarma.
\end{enumerate}

Análogamente, \texttt{stage\_equations.py} define \texttt{CONTROL\_LIMITS}, un diccionario que asigna a cada variable de control su rango operativo válido. La función \texttt{validate\_controls} verifica presencia, tipo numérico y rango de todas las variables antes de ejecutar el modelo:

\begin{verbatim}
for key, (vmin, vmax) in CONTROL_LIMITS.items():
    value = controls[key]
    if not isinstance(value, (int, float)):
        raise TypeError(...)
    if not math.isfinite(value):
        raise ValueError(...)
    if value < vmin or value > vmax:
        raise ValueError(...)
\end{verbatim}

Este patrón de validación en tres niveles (configuración $\to$ entrada $\to$ ejecución) garantiza que el modelo nunca opera con parámetros fuera de especificación, independientemente del origen de los datos.

\subsection{Núcleo Fenomenológico (\texttt{stage\_equations.py})}

Cada etapa del proceso se implementa como una función pura que recibe las variables de control y las especificaciones de equipo, y retorna un diccionario con los resultados. La firma genérica es:

\begin{verbatim}
def stage_n(controls: dict, specs: dict) -> dict:
    # balances de materia y energía
    return {"output_var": valor, ...}
\end{verbatim}

La función \texttt{run\_process\_model} orquesta la ejecución secuencial:

\begin{verbatim}
def run_process_model(controls, specs, limits):
    validate_controls(controls)
    s0 = stage_0(controls, specs)
    s1 = stage_1(controls, specs, s0)
    s2 = stage_2(controls, specs, s1)
    s3 = stage_3(controls, specs, s2)
    s4 = stage_4(controls, specs, s3)
    s5 = stage_5(controls, specs, s4)
    result = merge(s0..s5)
    indicators = enforce_capacity_constraints(
        controls, specs, limits, result)
    return {**result, **indicators}
\end{verbatim}

Cada función de etapa es independiente y puede ser modificada, reemplazada o probada de forma aislada. Para adaptar el DT a un nuevo proceso, solo es necesario reemplazar las funciones \texttt{stage\_n} y actualizar \texttt{CONTROL\_LIMITS} y \texttt{EQUIPMENT\_SPEC\_DEFAULTS}.

La inercia dinámica de primer orden se implementa como una función auxiliar independiente que puede aplicarse a cualquier variable:

\begin{verbatim}
def first_order_inertia(pv_old, sp, dt, tau):
    alpha = 1.0 - math.exp(-dt / tau)
    return pv_old + (sp - pv_old) * alpha
\end{verbatim}

\subsection{Motor de Restricciones (\texttt{equipment\_constraints.py})}

El motor de restricciones evalúa seis condiciones de capacidad sobre los resultados del modelo fenomenológico. Cada condición sigue el mismo patrón:

\begin{verbatim}
required = carga_solicitada
available = capacidad_nominal * factor_seguridad
indicators["util_pct"] = (required/available)*100
if required > available:
    issues.append({
        "code": "STAGEn_CAPACITY",
        "equipment": "...",
        "message": "...",
        "recommendation": "..."
    })
\end{verbatim}

Si hay incumplimientos, se lanza \texttt{EquipmentCapacityError} con la lista completa de issues. Este diseño desacopla la detección de fallas de la lógica del proceso: el modelo fenomenológico no necesita saber qué restricciones existen, y el motor de restricciones no necesita conocer los detalles del balance de materia.

\subsection{Interfaz Visual (\texttt{app.py}) y Bucle de Simulación}

La interfaz en Streamlit implementa un bucle temporal que avanza en pasos discretos ($\Delta t = 1\,\text{s}$). En cada paso:

\begin{enumerate}
    \item Lee los setpoints ingresados por el operador.
    \item Si hay cambios, ejecuta \texttt{run\_process\_model} para calcular el nuevo estado estacionario.
    \item Aplica la inercia de primer orden a cada variable para obtener el valor transitorio.
    \item Actualiza los indicadores y alarmas en la interfaz.
    \item Espera el próximo paso (simulado o en tiempo real).
\end{enumerate}

La visualización incluye: diagrama de flujo interactivo con balances de masa, KPIs en tarjetas numeradas, historial de variables versus setpoint y panel de alarmas FMEA. La comunicación sigue el estándar ISO 23247 [4] utilizando submodelos Asset Administration Shell (AAS, IEC 63278) [8] sobre OPC UA [7].

\subsection{Sistema de Pruebas Unitarias}

El directorio \texttt{tests/} contiene once pruebas que verifican:

\begin{enumerate}
    \item \textbf{Consistencia del balance de materia} (\texttt{test\_balance\_is\_consistent}): La masa total de salidas difiere de la masa de entrada en menos del $1\%$.
    \item \textbf{Rendimientos no negativos} (\texttt{test\_all\_rendimientos\_are\_non\_negative}): Todos los flujos másicos son $\geq 0$.
    \item \textbf{Caso base contra referencia} (\texttt{test\_baseline\_matches\_project\_final}): Con los parámetros nominales, los KPIs coinciden con los valores de referencia (ej. eficiencia de extracción $88\%$, polvo final $301.6\,\text{kg/h}$, rendimiento global $76.40\%$).
    \item \textbf{Detección de bloqueos} (\texttt{test\_dryer\_capacity\_blocks}, \texttt{test\_interchanger\_capacity\_too\_small\_blocks}, \texttt{test\_baseline\_capacity\_passes}): Especificaciones pequeñas disparan el error correcto; las nominales no disparan falsos positivos.
    \item \textbf{Validación de controles} (\texttt{test\_invalid\_control\_range\_raises}, \texttt{test\_non\_finite\_control\_raises}, \texttt{test\_outputs\_are\_finite\_and\_non\_negative}): Valores fuera de rango o no finitos son rechazados.
\end{enumerate}

Estas pruebas se ejecutan con \texttt{python -m unittest discover tests} y deben pasar antes de cualquier modificación al código, garantizando que no se introduzcan regresiones.

\section{Requisitos de Replicabilidad}

Para adaptar la arquitectura a una nueva industria de proceso, se deben seguir tres pasos, cada uno con un entregable de código específico:

\subsection{Paso 1: Mapeo de Equipos y Configuración}

Identificar los equipos del proceso y sus capacidades nominales. Crear un diccionario \texttt{EQUIPMENT\_SPEC\_DEFAULTS} con las capacidades de diseño y \texttt{CAPACITY\_LIMIT\_DEFAULTS} con los factores de seguridad operativos. La función \texttt{validate\_equipment\_specs} verificará que todos los valores estén dentro de rangos físicamente posibles. Este archivo es el único punto de contacto entre el código y la planta física.

\subsection{Paso 2: Implementación de Balances}

Para cada operación unitaria, implementar una función \texttt{stage\_n(controls, specs) -> dict} que resuelva los balances de materia y energía. Si no se requiere un modelo fenomenológico detallado, puede implementarse una versión simplificada con balances globales o factores de rendimiento empíricos.

\subsection{Paso 3: Definición de Variables de Control}

Definir \texttt{CONTROL\_LIMITS} con los rangos operativos de cada variable de control. Estas variables son los puntos de consigna que el operador puede modificar desde la interfaz. La función \texttt{validate\_controls} garantizará que ningún valor fuera de especificación alcance el modelo.

La arquitectura está diseñada para que estos tres pasos sean independientes: se puede cambiar la configuración de equipos sin modificar los balances, y viceversa.

\section{Caso de Validación: Producción de ISP}

Para validar la arquitectura, se instanció el DT para una planta piloto de Proteína Aislada de Soya (ISP) con capacidad nominal de $1000\,\text{kg/h}$. El proceso comprende seis etapas: molienda y mezcla de agua (Etapa~0), lixiviación alcalina (Etapa~1), pasteurización HTST (Etapa~2), concentración por ósmosis inversa y evaporación al vacío (Etapa~3), precipitación isoeléctrica y centrifugación (Etapa~4), y secado por atomización (Etapa~5).

\subsection{Verificación del Balance de Materia}

Con los parámetros nominales ($F_{\text{soya}} = 1000\,\text{kg/h}$, $Q_{\text{agua}} = 12\,\text{m}^3/\text{h}$, pH 8.5, temperatura secado $185\,^\circ\text{C}$), el modelo produce los siguientes KPIs, consistentes con las pruebas unitarias:

\begin{itemize}
    \item Eficiencia de extracción: $88.0\%$
    \item Proteína precipitada: $292.3\,\text{kg/h}$
    \item Polvo proteico final: $301.6\,\text{kg/h}$ ($286.5\,\text{kg/h}$ de proteína)
    \item Rendimiento global de proteína: $76.40\%$
    \item Humedad residual del polvo: $5.0\%$
\end{itemize}

El cierre del balance de materia global tiene un error inferior al $0.5\%$, validando la consistencia interna del modelo.

\subsection{Diagnóstico Estocástico del Diseño Base}

Se ejecutaron 10 batches independientes de Monte Carlo (100,000 corridas) sobre el diseño base para evaluar su robustez. Las 19 variables de control se perturbaron aleatoriamente dentro de rangos amplios. Solo el $4.70\%$ de las corridas fueron exitosas ($\sigma = \pm 0.18\%$, IC 95\%: $[4.59\%, 4.81\%]$). Los cuellos de botella identificados fueron el tanque de lixiviación TK-101 ($78.23\%$ de las corridas con falla), el tanque de agua TK-100 ($69.50\%$), el evaporador EV-301 ($57.06\%$) y el pasteurizador HX-201 ($40.69\%$).

\subsection{Redimensionamiento y Validación Final}

A partir del diagnóstico, se redimensionaron los equipos críticos en la configuración (Tabla \ref{tab:restrictions}). Tras la reconfiguración, una campaña de 30 batches (300,000 corridas) con rangos operativos ampliados ($\pm20\%$ en alimentación, $\pm25\%$ en flujo de agua) arrojó los siguientes resultados:

\begin{itemize}
    \item \textbf{Tasa de éxito:} $99.88\%$ (299,633 de 300,000 corridas exitosas).
    \item \textbf{Desviación estándar entre lotes:} $\pm 0.026\%$, con IC 95\%: $[99.87\%, 99.89\%]$.
    \item \textbf{Nivel Sigma:} $Z = 3.03\sigma$ (bruto), $4.53\sigma$ con desplazamiento $1.5\sigma$, equivalente a $\sim340$ DPMO.
    \item \textbf{Fallas residuales:} Solo \texttt{STAGE1\_TANK\_CAPACITY} (TK-101) en $0.12\%$, bajo condiciones extremas de residencia $>100$ minutos. Las cinco restricciones restantes no se activaron en ninguna corrida.
\end{itemize}

\subsection{Análisis Energético}

La Tabla \ref{tab:energy_comparison} muestra el impacto de integrar ósmosis inversa (OI) como etapa de preconcentración. La OI remueve $2718.3\,\text{kg/h}$ de agua en fase líquida con solo $8.5\,\text{kW}$ eléctricos, reduciendo la carga térmica del evaporador en $836.2\,\text{kW}$ ($19.5\%$). El ahorro neto equivalente es de $827.7\,\text{kW}$, lo que permite amortizar la inversión en la membrana en un periodo inferior a $1.5$ años.

\section{Conclusiones}

Se presentó la arquitectura computacional de un Gemelo Digital Fenomenológico diseñado para ser replicable en diferentes industrias de proceso. Las características clave que habilitan esta replicabilidad son: (i) configuración externalizada con validación automática, (ii) funciones de etapa independientes y reemplazables, (iii) motor de restricciones desacoplado del modelo fenomenológico, y (iv) pruebas unitarias que verifican la integridad del sistema.

La instanciación de la arquitectura para una planta de ISP demostró su utilidad práctica: el diagnóstico de Monte Carlo identificó los cuellos de botella del diseño base ($4.70\%$ de éxito), el redimensionamiento guiado por la arquitectura elevó la robustez a $99.88\%$ ($4.53\sigma$), y el análisis energético cuantificó un ahorro del $19.5\%$ mediante OI.

El código está estructurado para que cualquier ingeniero de procesos pueda adaptarlo siguiendo tres pasos: configurar los equipos en \texttt{equipment\_specs.py}, implementar los balances en \texttt{stage\_equations.py} y definir las variables de control en \texttt{CONTROL\_LIMITS}. El resto de la infraestructura (validación, restricciones, interfaz visual, pruebas) funciona sin modificaciones.
